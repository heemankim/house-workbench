# wb-plan: PRD 분석 및 작업 계획

PRD/요구사항을 분석하여 실행 가능한 작업 계획(PLAN.md)을 생성합니다.

## 입력
- `$ARGUMENTS`: Task 이름 (선택, 없으면 대화형)

## 핵심 원칙

**모호한 비즈니스 표현 → 구체적 데이터 기준**을 자연스럽게 도출합니다.

PRD는 비즈니스 언어로 작성됩니다. "완료되면", "유효한 경우", "적절한 시점에" 같은
표현은 코드로 옮기려면 반드시 **데이터 기준**이 필요합니다.
이 커맨드는 그 기준을 자연스럽게 끌어내는 대화 프로세스입니다.

---

## 프로세스

### 1. PRD 입력

```
📄 PRD를 입력해주세요.
   (텍스트 붙여넣기 / 파일 경로 / URL)
```

### 2. 흐름 정리

PRD를 읽고 **사용자 시나리오 흐름**으로 재구성합니다.
비즈니스 문서의 구조가 아닌, 실제 사용 흐름 순서로 정리합니다.

```markdown
## 흐름 정리

1. 사용자가 상품을 선택한다
2. 신청서를 작성한다
3. 신청을 제출한다
4. **신청이 완료되면** → 알림을 보낸다
5. 관리자가 승인한다
6. **승인이 완료되면** → 결과를 안내한다
```

**굵은 표시**는 다음 단계에서 기준 분석이 필요한 지점입니다.

### 3. 기준 도출 (핵심)

흐름에서 **조건이 포함된 문장**을 하나씩 짚어갑니다.
형식적인 질문이 아니라, 코드에서 실제로 필요한 판단 기준을 대화로 끌어냅니다.

#### 도출 방식

각 조건 문장에 대해 **3단계 질문**을 진행합니다:

**Step A. 현상 확인** — "이게 지금 어떻게 되고 있나요?"
```
💬 "신청이 완료되면 알림을 보낸다" 에서,

   현재 신청 완료는 어떻게 처리되고 있나요?
   (기존 로직이 있는지, 완전 신규인지)
```

**Step B. 기준 구체화** — "그러면 데이터로는 어떤 상태인가요?"
```
💬 신청 완료의 기준이 궁금합니다:

   [1] 사용자가 "제출" 버튼을 누른 시점
   [2] 서버에서 데이터 저장이 완료된 시점
   [3] 결제까지 완료된 시점
   [4] 다른 기준이 있나요?
```

사용자가 모르면 → **미확인으로 표시**하고 넘어갑니다.
이것 자체가 중요한 산출물입니다.

**Step C. 엣지케이스** — "이런 경우는 어떻게 하나요?"
```
💬 이런 경우를 확인하고 싶습니다:

   - 신청 후 결제 실패하면? → 완료로 볼 건지?
   - 중복 신청은 가능한지?
   - 신청 취소가 가능한 시점은?
```

#### 코드 탐색 병행 (Serena 활용)

대화 중 기존 코드베이스를 Serena 심볼릭 도구로 탐색합니다:

```
🔍 코드 탐색 (Serena):

1. 엔티티 구조 → get_symbols_overview / find_symbol
   src/application/entity.ts:
     status: 'DRAFT' | 'SUBMITTED' | 'COMPLETED' | 'CANCELLED'
   → "완료"는 status = 'COMPLETED' 시점으로 보입니다.

2. 관련 서비스 → find_referencing_symbols
   src/application/service.ts:complete()
   → 상태 변경 로직 존재

이 기준이 맞나요?
```

Serena 미활성화 시 Grep/Glob으로 탐색:
```
🔍 기존 코드에서 관련 로직을 찾지 못했습니다.
   → 신규 구현이 필요합니다. 데이터 모델부터 설계해야 합니다.
```

### 3.5. Serena 프로젝트 활성화 (자동)

코드 탐색 전 `skills/code-analysis/SKILL.md`의 **프로젝트 활성화** 프로시저를 실행합니다.
비활성화 또는 Serena MCP 미설치 시 → Grep/Glob fallback

### 4. 분석 결과 정리

모든 규칙을 표로 정리합니다:

```markdown
## 비즈니스 규칙

### Rule 1: 신청 완료 → 알림 발송

| 항목 | 비즈니스 | 데이터 기준 | 상태 |
|------|----------|------------|------|
| Trigger | 신청이 완료되면 | `status: SUBMITTED → COMPLETED` | ✅ 확인됨 |
| Condition | 정상 신청건 | `type IN ('NEW', 'RENEWAL')` | ✅ 확인됨 |
| Action | 알림을 보낸다 | 푸시? 이메일? | ❓ 미확인 |

**기존 코드:**
- `src/application/service.ts:complete()` — 상태 변경 로직 있음
- 알림 발송 로직 — 없음 (신규)

**엣지케이스:**
- 결제 실패 시: 완료 처리 안 함 (확인됨)
- 중복 신청: 불가 (확인됨)

**미확인:**
- [ ] 알림 채널 (푸시/이메일/SMS)
- [ ] 알림 발송 실패 시 재시도 정책
```

### 5. 태스크 분해

분석 결과에서 자연스럽게 태스크가 도출됩니다:

```markdown
## 태스크 분해

분석에서 도출된 작업 항목:

1. [📦 code] 신청 완료 이벤트 추가 (backend)
   ← Rule 1: Trigger 구현
   - status 변경 시 이벤트 발행
   - 기존 코드: service.ts:complete() 수정

2. [📦 code] 알림 발송 핸들러 (backend)
   ← Rule 1: Action 구현
   - 이벤트 수신 → 알림 발송
   - 신규 구현

3. [⏳ wait] 알림 채널 확인
   ← Rule 1: 미확인 사항
   - 기획팀 확인 필요

4. [📦 code] 알림 UI (frontend)
   ← depends: #3
```

**태스크 도출 기준:**
- ✅ 확인됨 → 바로 code 태스크
- ❓ 미확인 → wait 태스크 (확인 후 code로 전환)
- 신규 구현 → 설계 포함된 code 태스크
- 기존 수정 → 영향 범위 명시된 code 태스크

### 6. PLAN.md 생성

위 내용을 모두 포함한 PLAN.md를 생성합니다:

```markdown
data/tasks/{task-name}/
└── PLAN.md
    ├── 흐름 정리
    ├── 비즈니스 규칙 (Trigger/Condition/Action + 데이터 기준)
    ├── 미확인 사항 목록
    └── 태스크 분해
```

### 7. 다음 단계 안내

```
┌─────────────────────────────────────────────────┐
│ ✅ 분석 완료: {task-name}                        │
├─────────────────────────────────────────────────┤
│ 📋 규칙: 5개 (확인됨 3 / 미확인 2)              │
│ 📦 태스크: 6개 (실행 가능 4 / 대기 2)           │
│   └─ 독립 실행 가능: 3개 (병렬 실행 가능)       │
│ 📁 PLAN.md: data/tasks/{task-name}/PLAN.md      │
├─────────────────────────────────────────────────┤
│ 다음 단계:                                       │
│                                                  │
│ [1] /wb-start {task-name}                        │
│     → PLAN.md 기반으로 태스크 환경 세팅          │
│     → 이후 /wb-run으로 독립 서브태스크 병렬 실행 │
│                                                  │
│ [2] 미확인 사항 먼저 해결                        │
│     → 확인 후 /wb-plan으로 재분석                │
│                                                  │
│ [3] PLAN.md 직접 수정 후 /wb-start               │
└─────────────────────────────────────────────────┘
```

#### 파이프라인 실행

독립 서브태스크가 2개 이상이면 파이프라인 실행을 제안합니다:

```
💡 독립 서브태스크 3개가 병렬 실행 가능합니다.
   /wb-start → /wb-run 파이프라인으로 빠르게 진행할까요?

   [1] /wb-start {task-name} 후 수동 진행
   [2] /wb-start → /wb-run 자동 연결
       → 환경 세팅 후 바로 병렬 실행 시작
```

---

## wb-start 연동

wb-start 실행 시 PLAN.md가 있으면:
- 서브태스크를 PLAN.md의 태스크 분해에서 가져옴
- 사용자에게 확인 후 진행
- PLAN.md의 미확인 사항은 wait 타입 서브태스크로 자동 생성

## 기준 도출 팁

### 자주 모호한 비즈니스 표현

| 표현 | 물어봐야 할 것 |
|------|----------------|
| "~이 완료되면" | 완료의 데이터 기준 (상태값? 이벤트? 시간?) |
| "유효한 경우" | 유효성 판단 기준 (필드? 기간? 권한?) |
| "적절한 시점에" | 구체적 트리거 (사용자 액션? 배치? 스케줄?) |
| "관련된 사용자에게" | 대상 범위 (전체? 구독자? 권한 기반?) |
| "필요한 정보를" | 어떤 필드를, 어디서 가져와서 |
| "정상적으로 처리" | 성공 기준, 실패 시 동작 |
| "이전 데이터" | 몇 건? 어떤 기간? 마이그레이션 필요? |

### 기존 코드 탐색 전략 (Serena)

```
1. 엔티티/모델 먼저 → get_symbols_overview / find_symbol
2. 서비스 레이어 → find_symbol({Service}, depth=1)
3. 이벤트/핸들러 → find_referencing_symbols({Event})
4. API 엔드포인트 → search_for_pattern("@(Get|Post|Put|Delete)")
```

Serena 미사용 시:
```
1~4 동일 전략, Grep/Glob으로 탐색
```
