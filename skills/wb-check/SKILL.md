# wb-check: 설계-구현 검증

현재 서브태스크의 구현이 설계/요구사항과 일치하는지 검증합니다.

## 입력
- `$ARGUMENTS`: 없음 (현재 서브태스크 자동 감지)

## 프로세스

### 1. 현재 서브태스크 확인

```
📍 현재 서브태스크: PG 연동 (1/3)
📦 Task 유형: code
```

### 2. 비교 대상 수집

#### code 유형
```
설계/요구사항:
├─ TASK.md의 서브태스크 목표
├─ 관련 설계 문서 (있으면)
└─ 이전 의사결정 사항

구현:
├─ 변경된 파일들
├─ git diff 내용
└─ 새로 추가된 코드
```

#### ops 유형
```
요구사항:
├─ TASK.md의 서브태스크 목표
├─ 예상 결과물
└─ 사용할 도구/서비스

구현:
├─ artifacts/ 폴더 내용
├─ 생성된 워크플로우/설정
└─ 실행 결과
```

### 3. checker agent 호출

```
🤖 checker agent (opus) 호출 중...

비교 항목:
- API 엔드포인트
- 데이터 모델
- 비즈니스 로직
- 에러 처리
```

### 3.5. tester agent 호출 (code 유형만)

code 유형 서브태스크의 경우, checker 검증 후 테스트 커버리지를 점검합니다:

```
🤖 tester agent (sonnet) 호출 중...

점검 항목:
- 신규 코드 테스트 작성 여부
- 테스트 커버리지 (목표: >= 80%)
- 기존 테스트 회귀 감지
```

```

│ 🧪 테스트 평가                                           │

│  2. webhook 재전송 로직                                 │
```

### 5. Match Rate 기준 안내

```
📊 Match Rate 기준:
├─ >= 90%: ✅ 통과 → /wb-done 진행 가능
├─ 70-89%: ⚠️ 수정 권장 → Gap 해결 후 재검증
└─ < 70%: ❌ 재작업 필요 → 설계 재검토
```

### 5.5. 커버리지 기준 (code 유형)

```
📊 커버리지 기준:
├─ >= 80%: ✅ 통과
├─ 60-79%: ⚠️ 추가 테스트 권장
└─ < 60%: ❌ 테스트 작성 필요
```

### 6. 다음 단계 제안

#### Match Rate >= 90%
```
✅ 검증 통과!

다음 단계:
- /wb-done으로 서브태스크 완료
- /wb-pr로 PR 생성
```

#### Match Rate 70-89%
```
⚠️ 수정 권장

Gap 수정 후:
- /wb-check로 재검증
- 또는 /wb-done --skip-check로 강제 진행
```

#### Match Rate < 70%
```
❌ 재작업 필요

제안:
- TASK.md 요구사항 재확인
- 설계 문서 검토
- 누락된 기능 구현
```

## 옵션

| 옵션 | 설명 |
|------|------|
| `--detail` | 모든 비교 항목 상세 표시 |
| `--focus <area>` | 특정 영역만 검증 (api, model, logic) |
| `--against <file>` | 특정 설계 문서와 비교 |

### 예시

```bash
/wb-check --detail              # 상세 비교 결과
/wb-check --focus api           # API만 검증
/wb-check --against design.md   # 특정 문서와 비교
```
